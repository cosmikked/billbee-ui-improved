# Product Register

Use this register when design serves an active task: dashboards, settings, forms, admin screens, tables, workflows, and authenticated application UI.

## Core stance

The bar is earned familiarity. Users should trust the interface immediately and stay focused on the task. Surprise is expensive here.

## Typography

- System fonts are legitimate.
- One good sans family is often enough.
- Prefer fixed, product-oriented type scales over dramatic fluid heading systems.
- Keep stronger hierarchy for section structure, not decorative flourish.

## Color

- Default to `Restrained`.
- Reserve saturated accent color for primary action, selection, and meaningful state.
- Define semantic states cleanly: hover, focus, active, disabled, loading, error, warning, success, and info.

## Layout

- Familiar navigation and form patterns are features.
- Predictable grids and repeated structure improve speed and confidence.
- Responsive behavior should be structural, not decorative.

## Components

- Every interactive control needs complete states.
- Prefer skeleton states over generic spinners in content regions.
- Empty states should teach the next step.
- Keep component vocabulary consistent across screens.

## Motion

- Use short, state-driven motion.
- Avoid choreographed page-load animation.
- Motion should explain change, not decorate the screen.

## Product bans

- decorative motion without state meaning
- display fonts in buttons, labels, or dense UI
- inconsistent component styling across related screens
- reinvented standard affordances for flavor
- loud inactive states

## Product permissions

- system or highly familiar sans stacks
- dense layouts when the task needs them
- standard app-shell patterns
- consistency over novelty
